# Как я написал Telegram ID-бот с определением дата-центра и возраста аккаунта

Всё началось с того, что мне надоело пересылать сообщения в @userinfobot каждый раз, когда нужен чей-то Telegram ID. Пересылать, ждать ответ, копировать число — рутина, которая повторяется десятки раз в день, если ты пишешь ботов или администрируешь группы.

Существующие боты делают ровно одно действие: принимают пересланное сообщение и возвращают ID. Всё. Ни контекста, ни удобства, ни дополнительных данных. Я решил, что можно лучше — и написал своего бота, который не просто возвращает число, а даёт полную картину по аккаунту.

## Что получилось

Бот [@usergtidbot](https://t.me/usergtidbot) — «User Get ID». Определяет Telegram ID пользователей, групп и каналов. Но помимо самого ID, выдаёт дата-центр, примерную дату регистрации, бейдж возраста аккаунта, историю юзернеймов. Есть inline-режим, QR-коды, сравнение аккаунтов, экспорт в JSON и CSV, реферальная система.

Дальше расскажу про технические решения, которые показались мне интересными.

## Нативные кнопки выбора вместо пересылки

В 2023 году Telegram добавил `KeyboardButtonRequestUsers` — кнопку, которая открывает системный пикер контактов. Пользователь нажимает кнопку, выбирает любого человека из списка контактов, и бот получает его ID напрямую.

```python
from telegram import KeyboardButton, KeyboardButtonRequestUsers, ReplyKeyboardMarkup

button = KeyboardButton(
    text="Выбрать пользователя",
    request_users=KeyboardButtonRequestUsers(
        request_id=1,
        user_is_bot=None,  # любые аккаунты
        max_quantity=1
    )
)
markup = ReplyKeyboardMarkup([[button]], resize_keyboard=True)
```

Это принципиально лучше пересылки сообщений по нескольким причинам:

1. **Не нужно искать сообщение от нужного человека** — просто выбираешь из контактов
2. **Работает для людей, с которыми нет общих чатов** — если они в контактах телефона
3. **Не зависит от настроек приватности** — при пересылке сообщения поле `forward_from` может отсутствовать, если пользователь запретил это в настройках. С нативной кнопкой этой проблемы нет

Есть аналогичные кнопки для выбора групп (`KeyboardButtonRequestChat`) — это покрывает все типы чатов.

Большинство ID-ботов до сих пор просят пересылать сообщения. Видимо, авторы либо не обновляли код, либо не следят за обновлениями Bot API. А зря — UX с кнопками кардинально отличается.

## Определение дата-центра через file_id

Это, пожалуй, самая интересная техническая часть проекта. Когда Telegram возвращает фотографию профиля пользователя, в `file_id` закодирована информация о дата-центре, где хранится файл. А дата-центр файла — это дата-центр аккаунта.

Идея не моя — метод известен в сообществе. Но реализация потребовала разбора бинарного формата `file_id`.

Telegram использует base64-кодированную структуру для file_id. Номер дата-центра находится в определённом байте:

```python
import base64

def extract_dc(file_id: str) -> int | None:
    try:
        data = base64.urlsafe_b64decode(file_id + "==")
        if len(data) < 8:
            return None
        dc_id = data[4]
        if dc_id in (1, 2, 3, 4, 5):
            return dc_id
        return None
    except Exception:
        return None
```

Пять дата-центров Telegram на момент написания:
- **DC1** — Майами (старый, для ранних аккаунтов)
- **DC2** — Амстердам (Европа и СНГ)
- **DC3** — Майами (расширение DC1)
- **DC4** — Амстердам (второй европейский)
- **DC5** — Сингапур (Азия)

Дата-центр назначается при регистрации на основе номера телефона. Европейские и российские номера обычно попадают на DC2 или DC4, азиатские — на DC5, американские — на DC1 или DC3. Но это не жёсткое правило: бывают исключения, особенно у аккаунтов, зарегистрированных на виртуальные номера.

Если у пользователя нет фотографии профиля, дата-центр определить нельзя — file_id отсутствует. Это ограничение метода, обойти его пока никак. Можно попробовать запросить другие файлы, связанные с аккаунтом (стикеры в описании и т.п.), но на практике аватарка — самый надёжный источник.

Знание дата-центра бывает полезно при отладке. Если сообщения доходят с задержкой, а пользователь на DC5, а ваш бот-сервер в Европе — это может объяснять лаг. Не то чтобы вы можете что-то с этим сделать, но хотя бы понятно, где искать причину.

## Оценка даты регистрации по ID

Telegram не предоставляет дату регистрации аккаунта через API. Но IDs назначаются последовательно (с незначительными отклонениями), и за годы существования Telegram сообщество накопило достаточно реперных точек.

Я собрал таблицу диапазонов ID с привязкой к датам. Источники — публичные исследования, скриншоты ранних аккаунтов, посты на форумах. Выглядит примерно так:

```python
ID_RANGES = [
    (1,         "~2013-08"),  # первые аккаунты, включая тестовые
    (1000000,   "~2013-10"),
    (10000000,  "~2014-01"),
    (50000000,  "~2014-07"),
    (100000000, "~2015-02"),
    (200000000, "~2016-01"),
    (300000000, "~2016-08"),
    (400000000, "~2017-03"),
    (500000000, "~2018-01"),
    (700000000, "~2018-09"),
    (900000000, "~2019-05"),
    (1100000000, "~2020-02"),
    (1300000000, "~2020-10"),
    (1500000000, "~2021-04"),
    (1800000000, "~2021-10"),
    (2100000000, "~2022-04"),
    (5000000000, "~2023-02"),  # заметный скачок в нумерации
    (6000000000, "~2023-09"),
    (7000000000, "~2024-06"),
    # ...дополняется по мере появления данных
]
```

Обратите внимание на скачок между 2.1 и 5 миллиардами. Telegram перешёл на 64-битные ID и, судя по всему, сдвинул стартовую точку. Из-за этого оценки в районе 2022-2023 менее точны — между реперными точками большой разрыв.

Точность оценки — плюс-минус пара месяцев. Для ранних аккаунтов (2013-2015) точность выше, потому что пользователей было меньше и рост был более предсказуемым. Для недавних — разброс больше из-за массовых регистраций.

На основе оценочной даты бот присваивает бейджи:

- **OG** — 2013-2014. Те, кто был в Telegram с самого начала
- **Ветеран** — 2015-2017
- **Опытный** — 2018-2020
- **Обычный** — 2021-2023
- **Новичок** — 2024 и позже

Бейджи — это просто fun element, но люди неожиданно часто ими делятся. «О, я OG!» — и скриншот в чат. Кто-то целенаправленно проверяет друзей, чтобы найти «самого раннего». Геймификация работает даже в таких мелочах.

## Отслеживание юзернеймов

Каждый раз, когда бот видит пользователя (через lookup или взаимодействие), он записывает текущий username в базу с таймстампом. Если при следующем обращении username изменился, бот фиксирует новое значение.

Со временем накапливается история:
```
@john_doe (до 2025-03-15)
@johndoe (до 2025-09-22)
@j_doe (текущий)
```

Храню в SQLite — для такой задачи нет смысла поднимать что-то сложнее. Таблица простая: `user_id`, `username`, `first_seen`, `last_seen`.

```python
def update_username(user_id: int, username: str | None):
    current = db.execute(
        "SELECT username FROM usernames WHERE user_id = ? ORDER BY last_seen DESC LIMIT 1",
        (user_id,)
    ).fetchone()
    
    if current and current[0] == username:
        db.execute(
            "UPDATE usernames SET last_seen = ? WHERE user_id = ? AND username = ?",
            (now(), user_id, username)
        )
    else:
        db.execute(
            "INSERT INTO usernames (user_id, username, first_seen, last_seen) VALUES (?, ?, ?, ?)",
            (user_id, username, now(), now())
        )
```

Очевидное ограничение: бот видит только тех пользователей, которые с ним взаимодействуют. Это не всевидящий сканер — просто логирование того, что проходит через бота. Для полноценного трекинга нужен userbot с доступом к MTProto API, но это совсем другая история (и юридически скользкая).

## Inline-режим

Inline-режим оказался на удивление популярным. Пишешь `@usergtidbot <username>` прямо в любом чате — и получаешь карточку с ID, не покидая разговор. Для тех, кто часто работает в группах, это экономит заметное количество времени.

Для реализации используется `InlineQueryHandler`. Бот получает запрос, ищет пользователя и формирует `InlineQueryResultArticle`:

```python
async def inline_query(update: Update, context):
    query = update.inline_query.query.strip()
    if not query:
        return
    
    results = [
        InlineQueryResultArticle(
            id=str(uuid4()),
            title=f"ID: {user_id}",
            description=f"@{username} • DC{dc} • {age_badge}",
            input_message_content=InputTextMessageContent(
                message_text=format_user_card(user_data)
            )
        )
    ]
    await update.inline_query.answer(results, cache_time=60)
```

`cache_time=60` — результат кэшируется на минуту. Если поставить 0, Telegram будет слать запросы на каждый символ, что создаёт ненужную нагрузку. Я экспериментировал с разными значениями: 300 секунд слишком много (люди видят устаревшие данные), 10 секунд — слишком мало (нагрузка). Минута — разумный компромисс.

## Сравнение аккаунтов и групповые инструменты

Не самая очевидная фича, но оказалась востребованной. Выбираешь двух пользователей, и бот показывает, кто зарегистрировался раньше и на сколько.

Простое сравнение ID: меньший ID = более ранняя регистрация. Добавил разницу в оценочных датах и визуализацию:

```
Пользователь A: ~2016-03 (Ветеран)
Пользователь B: ~2022-11 (Обычный)
Разница: ~6 лет 8 месяцев
Раньше зарегистрирован: Пользователь A
```

Для администраторов групп сделал инструменты массовой работы: получить список всех админов с их ID и DC, экспортировать в CSV. Когда у тебя 10+ групп и нужно быстро проверить, кто есть кто — ручной поиск занимает вечность.

## Экспорт данных

JSON-экспорт сделал в первую очередь для себя — удобно прокидывать данные в скрипты. Потом добавил CSV для тех, кому проще работать в таблицах.

Формат JSON:
```json
{
    "user_id": 123456789,
    "username": "johndoe",
    "dc_id": 2,
    "dc_location": "Amsterdam",
    "estimated_reg": "~2018-03",
    "age_badge": "Опытный",
    "username_history": ["john_doe", "johndoe"]
}
```

Для группового экспорта (список админов группы, например) CSV оказался более востребованным. Люди экспортируют список, открывают в Google Sheets, фильтруют — стандартный рабочий процесс.

## Архитектура

Стек минималистичный:

- **python-telegram-bot** — асинхронная обёртка над Bot API. Зрелая библиотека, хорошая документация, активное комьюнити
- **SQLite** — для хранения юзернеймов, реферальных связей и статистики. При текущих объёмах нет причин усложнять
- **Polling** — бот работает в long polling режиме. Для однозадачного бота на одном сервере webhook не даёт преимуществ, а polling проще в отладке
- Никакого Docker, никакого Kubernetes — запускается через `nohup` и работает. Для бота с одним процессом и SQLite-базой это адекватный уровень сложности

Реферальная система хранит связи в той же SQLite: кто кого привёл, когда, сколько рефералов у каждого пользователя. При старте бот проверяет deep link параметр и записывает реферала, если пользователь новый.

## Что бы я сделал иначе

**Хранение username-истории в PostgreSQL.** SQLite работает, но при росте базы запросы с `ORDER BY last_seen` начнут тормозить. Индексы помогают, но это тот момент, когда проще сразу заложить нормальную БД. С другой стороны, SQLite до сих пор справляется, и YAGNI пока побеждает.

**Rate limiting с самого начала.** Добавил позже, когда заметил подозрительные паттерны запросов — кто-то явно скриптовал бота. Лучше бы сделал сразу: несколько строк кода на старте экономят часы разбирательств потом.

**Кэширование аватарок для DC-определения.** Сейчас каждый запрос идёт за фотографией профиля через API. При массовых запросах это узкое место. Кэш на 5-10 минут снял бы нагрузку — аватарки не меняются каждую минуту.

**Тесты.** Классическая история — начинал как мелкий pet project, тесты казались избыточными. Теперь, когда фич стало больше десятка, рефакторить страшно. Один неосторожный коммит — и inline-режим ломает экспорт. Добавлю, когда буду переписывать storage layer.

**Миграция на aiogram.** python-telegram-bot отлично работает, но aiogram больше распространён в русскоязычном сообществе, и для моих других проектов я уже перешёл на aiogram 3.x. Единый стек упростил бы поддержку.

## Итого

Бот решает одну понятную задачу — определение Telegram ID — но делает это с контекстом. Дата-центр, возраст аккаунта, история юзернеймов — всё это информация, которая уже есть в Telegram, просто не отображается в интерфейсе. Бот собирает её в одном месте и выдаёт за секунду.

Проект остаётся pet project, но с живой аудиторией и регулярным использованием. Иногда самые полезные инструменты — самые простые.
